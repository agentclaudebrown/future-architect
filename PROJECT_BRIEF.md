## WHAT THIS PROJECT IS

A white paper — and the body of work that follows from it — defining what the enterprise
IT architect role looks like in the age of agentic AI. Written by a practitioner, for
practitioners and the executives who deploy them.

The white paper stands alone as a public thought leadership document. It is also the
door into a larger internal Kyndryl initiative: rebuilding a formalised architecture
methodology for the AI age — the successor to IBM Architectural Thinking.

**The arc:**
1. White paper — establishes the philosophy, makes the argument, creates the conversation
2. Framework — defines the method: every phase, every agent, every output, standardised
3. Toolkit — pre-built agents, templates, workflows, artefacts
4. Training programme — certifies Kyndryl architects in the method
5. Managed service offering — packages the run-state agent fleet as a differentiated product

This brief covers Phase 1 (the white paper) and the governance structure for all phases.

---

## WHAT THIS PROJECT IS NOT

- A product spec or technical build (yet)
- A Kyndryl-only internal document — the white paper must stand alone publicly
- A breathless AI hype piece — it is a practitioner's argument, grounded in evidence
- A survey of AI tools — it is about the human role, not the technology catalogue
- Finished — this is an active, evolving body of work

## WORKFLOW

### The Three Roles

**Chris (Author / Orchestrator)**
Owns the project. Provides the voice, the judgment, the lived experience.
No AI writes the white paper — Chris writes it. AI assists.

**Claude Sonnet (Research / Development Partner)**
Working session environment. Research, brainstorming, thesis development,
structural decisions, capturing to SEEDS.md, maintaining project files.
New sessions should read SEEDS.md and WHITEPAPER_SKELETON.md first to rebuild context.

**Opus Writing Assistant (To Be Established)**
Dedicated Claude Opus instance. Writing assistant and test-reader.
Reviews drafts, stress-tests arguments, checks flow. Does NOT write sections.
Does NOT make structural decisions.

### The Working Session Pattern

1. Load context: read SEEDS.md tail and WHITEPAPER_SKELETON.md
2. Identify session goal: research, brainstorm, or draft review
3. Do the work
4. Save anything new to SEEDS.md
5. Update WHITEPAPER_SKELETON.md VERSION LOG if structural decisions made
6. Update DECISION_LOG.md with significant calls
7. Push changes to GitHub

### When to Use Which Tool

| Task | Tool |
|---|---|
| Researching a topic | Claude Sonnet (web search) |
| Developing an argument | Claude Sonnet |
| Capturing a new idea | Claude Sonnet → SEEDS.md |
| Drafting a section | Chris writes, then Opus reviews |
| Stress-testing a claim | Opus writing assistant |
| Structural decisions | Claude Sonnet + Chris judgment |
